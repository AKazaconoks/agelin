## 401 vs. 403: The Spec-Defined Distinction

**RFC 9110** (the current authoritative HTTP spec) draws the line cleanly, and the actor model maps directly to the authentication/authorization separation.

---

### What Each Code Actually Means

**401 Unauthorized — subject is the *request's credentials***

The spec (§15.5.2) says: "the request has not been applied because it **lacks valid authentication credentials** for the target resource."

- The server cannot yet make an access decision because it doesn't know who you are (or your credentials are invalid/expired).
- The name "Unauthorized" is a **spec-acknowledged misnomer** — 401 actually means **unauthenticated**.
- The spec requires (`MUST`) a `WWW-Authenticate` response header telling the client *how* to obtain valid credentials. No header → non-conformant.

**403 Forbidden — subject is the *server's policy decision***

The spec (§15.5.4) says: "the server understood the request but **refuses to fulfill it**."

- Identity is established (or the server has decided to make a final ruling regardless); the answer is simply *no*.
- This is an **authorization** decision — your role/group doesn't permit access.
- No `WWW-Authenticate` header is required or expected, because no credential change can fix this. The user would need a role change (an out-of-band operation).

| Dimension | 401 | 403 |
|---|---|---|
| Subject | The request's credentials | The server's access policy |
| Auth state | Unauthenticated / invalid creds | Authenticated |
| Authz state | Not evaluated yet | Evaluated and denied |
| `WWW-Authenticate` | MUST include | Not required |
| Actionable? | Yes — provide credentials | Typically no |

---

### Your Two Scenarios

**Not logged in → 401**
The request lacks proof of identity. Server cannot evaluate authorization yet. Must include `WWW-Authenticate`.

*Practical note for browser apps:* Most web apps return **302 → login page** instead of a raw 401, because browsers render 401s with a native credential dialog, not your custom login UI. This is a widely accepted convention that deviates from strict HTTP semantics. For REST APIs consumed programmatically, 401 is correct.

**Logged in but wrong role/group → 403**
Identity is established. The server has made a deliberate, terminal refusal. No credential change will help.

---

### Information Disclosure Nuance

The spec explicitly sanctions a third option: **404 Not Found**.

§15.5.4 says directly: "If the server does not wish to make this information available to the client, it can instead use **404 (Not Found)**."

**403 confirms the resource exists.** You can't forbid access to something that doesn't exist — returning 403 tells an attacker "this URL is real." If resource existence is itself sensitive (e.g., `/users/john-doe/records`, private repositories), 404 is the security-conscious choice. GitHub does exactly this for private repos.

---

### Decision Tree

```
Request arrives
│
├── Resource doesn't exist → 404
│
├── Resource exists but its existence is sensitive → 404 (spec-sanctioned)
│
├── No credentials / invalid credentials → 401 + WWW-Authenticate
│
└── Valid credentials, identity established
    ├── Role permits access → 200
    └── Role denies access → 403
```

---

### Most Common Mistake

Using **403 for unauthenticated requests**. The spec is unambiguous: 403 is a terminal refusal against a *known identity*. An unauthenticated request gets 401 (with a challenge), not 403.
