## 401 vs 403: The Actor Model

The distinction maps directly onto **authentication** vs **authorization** — two different questions:

| Code | Question | Implication |
|---|---|---|
| **401 Unauthorized** | *Who are you?* | Identify yourself; access may be granted with valid credentials |
| **403 Forbidden** | *Are you allowed?* | Identity is known (or irrelevant); access is denied regardless |

---

### 401 — Authentication failure

RFC 9110 §15.5.2: the request **lacks valid authentication credentials**. The name "Unauthorized" is a historical misnomer — the spec is explicit that this is about *authentication*, not authorization.

**Mandatory:** The response **MUST** include a `WWW-Authenticate` header specifying the challenge scheme (e.g., `Bearer realm="api"`). A 401 without it is non-conformant.

**Semantics:** *"We don't know who you are. Provide credentials and we may reconsider."* It is an **invitation to authenticate**.

Use when: no credentials were sent, or credentials are expired/invalid/malformed.

---

### 403 — Authorization failure

RFC 9110 §15.5.4: the server **understood the request but refuses to fulfill it**. If credentials were provided, the spec says the server considers them *insufficient* — re-sending the same credentials won't help.

**No** `WWW-Authenticate` required.

**Semantics:** *"We know who you are (or it doesn't matter). You are not permitted here."* It is a **final decision**.

Use when: the user is authenticated, but their role/group/ACL doesn't grant access.

---

### Mapping to your two scenarios

| Scenario | Correct code |
|---|---|
| User is **not logged in** (no/invalid credentials) | **401** |
| User is **logged in but lacks permission** (wrong group/role) | **403** |

The common mistake is returning 401 for all access failures. That's semantically wrong for an authenticated user — it incorrectly tells them "log in again," when logging in again won't help.

---

### Key nuances

**Information leakage:** 403 confirms the resource *exists*. If that's sensitive, RFC 9110 explicitly permits using **404** instead — a deliberate security choice, not a protocol violation.

**API vs browser clients:** Browser apps commonly redirect (302) to a login page for 401s. This is fine for UX, but APIs should return JSON 401/403 — redirecting API clients to an HTML login page is a common failure mode (they get `200 OK` with HTML they can't parse).

**OAuth 2.0 / RFC 6750** maps cleanly: `error="invalid_token"` → 401; `error="insufficient_scope"` → 403.

**Expired session:** The credential exists but is invalid server-side → **401** (authentication failure, even if identity was previously known).
