## 401 vs 403: The Actor Model

The key question is: **would providing (or changing) credentials change the outcome?**

### 401 Unauthorized — RFC 7235 §3.1

> "The request has not been applied because it lacks valid authentication credentials for the target resource."

- The server **does not know who you are**
- It is a **challenge**, not a rejection — the mandatory `WWW-Authenticate` header tells the client how to authenticate
- Semantics: "I don't know you — please identify yourself and retry"
- The name is a well-known misnomer; **401 means *unauthenticated*, not unauthorized**

**Use when:** the user is not logged in, has no token, or has an expired/invalid session.

### 403 Forbidden — RFC 7231 §6.5.3

> "The server understood the request but refuses to authorize it."

- The server **knows who you are** (or doesn't care), and the answer is still no
- It is a **definitive refusal** — no `WWW-Authenticate` challenge is issued because re-authenticating won't help
- The spec explicitly states: "the server considers [provided credentials] insufficient to grant access"

**Use when:** the user *is* authenticated but lacks the required role/group membership.

---

### Decision Table

| Scenario | Response |
|---|---|
| Not logged in at all | **401** — challenge to authenticate |
| Logged in, wrong group/role | **403** — known identity, insufficient privilege |
| Resource existence is confidential | **404** — spec-sanctioned alternative to 403 |

### The Clean Mental Model

Despite the confusing label, the RFCs describe:
- **401** = *unauthenticated* (identity unknown)
- **403** = *unauthorized* (identity known, permission denied)

The test: if switching to a different authenticated account *could* grant access, the root issue is authentication → **401**. If no plausible credential change would help → **403**.
