## The Core Distinction

The key is understanding **who** the server is talking to:

- **401** says: *"I don't know who you are. Tell me, and I might let you in."*
- **403** says: *"I know who you are (or don't care). You are not allowed. Authenticating will not help."*

---

## 401 Unauthorized — RFC 9110 §15.5.2

> "The 401 status code indicates that the request has not been applied because it lacks valid authentication credentials for the target resource. The server MUST send a `WWW-Authenticate` header field..."

**Key characteristics:**
1. **`WWW-Authenticate` header is mandatory** — without it, the response is malformed per spec. It tells the client *how* to authenticate (`Bearer`, `Basic`, etc.).
2. **It is an invitation to authenticate.** The implicit contract: try again with valid credentials and you *might* get in.
3. **The name is a misnomer.** "Unauthorized" actually means *unauthenticated* — the spec authors acknowledged this historical naming error.

**Use when:** User is not logged in, or sent credentials that are expired/invalid/malformed.

---

## 403 Forbidden — RFC 9110 §15.5.4

> "...A 403 response is not a challenge for credentials; it simply states that the server is refusing to authorize the request. **Authorization will not help**, and the request SHOULD NOT be repeated."

**Key characteristics:**
1. **No `WWW-Authenticate` header** — no invitation to authenticate.
2. **Identity is known or irrelevant.** The decision is final for this request.
3. **Confirms the resource exists.** If you want to hide existence, the spec explicitly permits substituting **404** — a common security pattern for admin endpoints.

**Use when:** User is authenticated but lacks the required role/group/permission.

## Applied to Your Scenario

| Scenario | Correct Response |
|---|---|
| User is not logged in (no credentials) | **401** |
| User is logged in but wrong role/group | **403** |
| User sent an expired/invalid token | **401** |
| User is on a blocklist regardless of credentials | **403** |

Your scenario conflates two distinct cases — "not logged in" → 401, "wrong group" → 403.

## The Actor Model, Precisely

The spec involves three actors: **client**, **authentication mechanism**, and **authorization policy**.

- **401** = the *authentication mechanism* is unsatisfied. The server can't identify the caller. Gating concern: *"Who are you?"*
- **403** = the *authorization policy* has ruled. Identity may be fully established, but the policy says no. Gating concern: *"Are you allowed?"*

---

## Real-World Note

Many frameworks return 401 for both cases — technically incorrect but widespread. The `WWW-Authenticate` requirement is the practical enforcement: if your response doesn't include that header, it should not be a 401.
