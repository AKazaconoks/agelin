## 401 vs. 403: The Actor Model

The two codes answer **different questions** at different layers:

| Code | Question | Layer |
|------|----------|-------|
| **401 Unauthorized** | *Who are you?* I can't act until I know. | Authentication |
| **403 Forbidden** | *I know who you are (or don't need to), and the answer is still no.* | Authorization |

The name "401 Unauthorized" is a well-known misnomer — RFC 9110 consistently uses the word *authentication* when defining it. Think of it as "401 Unauthenticated."

---

## What RFC 9110 Says

**§15.5.2 — 401 Unauthorized:**
> "The 401 status code indicates that the request has not been applied because it **lacks valid authentication credentials** for the target resource."

A 401 response **must** include a `WWW-Authenticate` header telling the client how to authenticate (e.g., `WWW-Authenticate: Bearer realm="myapp"`). This is not cosmetic — it is part of the HTTP challenge/response protocol.

**§15.5.4 — 403 Forbidden:**
> "The server understood the request but **refuses to fulfill it**... The client **SHOULD NOT** automatically repeat the request with the same credentials."

That "SHOULD NOT repeat" clause is the key signal: the problem is not *which* credentials, it is *what those credentials are allowed to do*. Re-authenticating as the same user won't help.

---

## Concrete Cases

**User is not logged in (no session/token):**
→ **401 Unauthorized.** No identity has been established. The server says "try again with credentials."

**User is logged in but lacks the required role/group:**
→ **403 Forbidden.** Identity is known; permission is denied. No amount of re-authenticating as the same user changes the answer.

---

## The Information Disclosure Nuance

Returning 403 implicitly confirms the resource exists. If that fact itself must be hidden (a draft document, a private admin endpoint, a user record that shouldn't be enumerable), **404 Not Found** is the safer choice — it reveals nothing about existence.

| Response | Discloses resource exists? | When to use |
|----------|---------------------------|-------------|
| 401 | Yes (implicitly — why challenge otherwise?) | Resource exists, requires login |
| 403 | Yes | Resource exists, wrong role/group, existence is not sensitive |
| 404 | No | Resource existence must be hidden from this actor |

This is a security architecture decision, not mandated by RFC 9110 — but it is well-established in security practice (e.g., OWASP guidance).
