Here is a precise breakdown of **401 vs 403**, based on the HTTP spec (RFC 9110):

---

## The Actor Model

The key distinction is the two-stage pipeline:

```
Request → [Authentication] → [Authorization] → Resource
               |                    |
              401                  403
         "Who are you?"      "I know who you are,
                              but you can't come in."
```

---

## 401 Unauthorized — "Unauthenticated"

RFC 9110 §15.5.2:
> "The request has not been applied because it **lacks valid authentication credentials** for the target resource."

The name is a well-known misnomer. "Unauthorized" really means **unauthenticated**. The RFC authors acknowledged this but kept the name for backward compatibility.

**The server MUST send a `WWW-Authenticate` header** with at least one challenge — this is the mechanical expression that 401 is an *invitation to authenticate*:
```
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="api", error="invalid_token"
```

Use 401 when:
- There are no credentials at all, OR
- The credentials are invalid (expired token, wrong password), AND
- Providing valid credentials *might* grant access

---

## 403 Forbidden — "Authorized but denied"

RFC 9110 §15.5.4:
> "The server understood the request but **refuses to fulfill it**."
> "If authentication credentials were provided, the server considers them **insufficient to grant access**. The client **SHOULD NOT** automatically repeat the request with the same credentials."

Use 403 when:
- The server has identified who the client is, AND
- That identity is denied regardless — retrying with the same credentials is futile

---

## The Precise Decision Table

| Dimension | 401 | 403 |
|---|---|---|
| Identity established? | No (or invalid) | Yes (or irrelevant) |
| Would valid credentials help? | Yes, possibly | No |
| Client should retry? | Yes, with new credentials | No |
| `WWW-Authenticate` required? | **Yes (MUST)** | No |
| Analogy | Bouncer asking for ID | Bouncer: "you're on the no-entry list" |

---

## Applied to Your Two Cases

**"User is not logged in"** → **401**. No authenticated identity. The response invites authentication. (Web apps often redirect to a login page instead — a pragmatic deviation from strict RFC conformance for browser flows.)

**"User is logged in but lacks privilege (wrong group/role)"** → **403**. The server knows who they are; their identity simply doesn't have access. No re-authentication as the same user will help.

---

## Bonus: The 404 Security Trade-off

For resources that should be hidden from unauthorized users, some systems deliberately return **404** ("Not Found") even when the resource exists. This prevents information leakage about the resource's existence. It's a deliberate security choice, not a spec violation.
