## 401 vs 403: The Actor Model

The distinction is **authentication vs. authorization** — i.e., *identity* vs. *permission*.

### 401 Unauthorized — "Who are you?"

- **RFC 7235 §3.1 / RFC 9110**: The request lacks valid authentication credentials for the target resource.
- The server **does not know who the client is** (anonymous, or credentials are invalid/missing).
- The server **MUST** include a `WWW-Authenticate` header with at least one challenge.
- This is **recoverable**: retry with valid credentials.

### 403 Forbidden — "I know who you are, but no."

- **RFC 9110 §15.5.4**: The server understood the request but refuses to authorize it.
- The server **knows the client's identity** but the client lacks sufficient privileges for this resource.
- Re-authenticating with the same credentials **will not help**.
- No `WWW-Authenticate` header is expected.
- This is **non-recoverable** for that user (without a privilege change).

---

## For Your Two Cases

| Scenario | Response |
|---|---|
| User is **not logged in** | **401** — prompt them to authenticate |
| User is logged in but **wrong group** | **403** — they're known, but unauthorized |

---

## Bonus: 404 as a Security Option

RFC 9110 explicitly permits a server to return **404 Not Found** instead of 403 if revealing that a resource *exists* is itself a security concern (e.g., hiding admin pages from unauthorized users). This is a deliberate trade-off of transparency for security.

---

**TL;DR**: 401 = unknown identity (authenticate first); 403 = known identity, insufficient privilege (no amount of re-logging-in will help without a role change).
