## 401 Unauthorized vs. 403 Forbidden

The fundamental distinction is **authentication** vs. **authorization**.

### 401 Unauthorized (RFC 7235 §3.1)

> "The 401 status code indicates that the request has not been applied because it **lacks valid authentication credentials** for the target resource."

- The server does not know *who* you are.
- The server **must** respond with a `WWW-Authenticate` header issuing a challenge.
- Re-authenticating (supplying credentials) could grant access.
- **Use when**: user is not logged in.

### 403 Forbidden (RFC 9110 §15.5.4)

> "The server understood the request but **refuses to authorize it**. If authentication credentials were provided, the server considers them insufficient to grant access."

- The server knows *who* you are, but you don't have permission.
- No `WWW-Authenticate` challenge is required or implied.
- Re-authenticating with the same credentials will not help.
- **Use when**: user is logged in but belongs to the wrong group / lacks the required role.

---

### The Actor Model in One Line

| Status | Question the server is asking |
|--------|-------------------------------|
| **401** | "Who are you?" (identity unknown) |
| **403** | "I know who you are — and the answer is no." (identity known, access denied) |

---

### Practical Decision Tree

```
Resource exists?
  └─ No → 404 (or 404 to hide existence)
  └─ Yes → Is the user authenticated (logged in)?
              └─ No → 401 (with WWW-Authenticate header)
              └─ Yes → Does the user have the required role/permission?
                          └─ No → 403
                          └─ Yes → 200
```

### Bonus: 404 as a Privacy Shield

Some systems intentionally return **404** for protected resources to avoid leaking that the resource even exists to unauthenticated/unauthorized requesters. This is valid as a security posture but violates strict HTTP semantics — use it deliberately and document it.
