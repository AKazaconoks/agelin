## 401 Unauthorized

- **Semantic**: "I don't know who you are."
- **Applies when**: The request lacks valid authentication credentials.
- **Required side effect**: The server **must** send a `WWW-Authenticate` header with at least one challenge.
- **Implication**: Supplying (or re-supplying) credentials *could* resolve the problem.
- **Actor model**: Anonymous / unidentified actor.

## 403 Forbidden

- **Semantic**: "I know who you are, but you're not allowed."
- **Applies when**: The server understood the request but refuses to authorize it — credentials are present but insufficient.
- **Key distinction**: Re-authenticating as the same user will **not** help. The refusal is a definitive authorization decision.
- **Actor model**: Identified actor with insufficient privileges.

---

## The Distinguishing Axis

| Scenario | Correct code | Reason |
|---|---|---|
| Not logged in (anonymous) | **401** | Identity is unknown; authenticate first |
| Logged in, wrong group/role | **403** | Identity is known; authorization fails |

The actor model in one sentence: **401 = authentication gap; 403 = authorization gap**.

---

## Information Disclosure Nuance

Returning **403** confirms the resource *exists* (just not accessible to the caller). If you want to hide resource existence from unauthorized users, some servers return **404** instead — this is a deliberate security-through-obscurity trade-off, not a spec requirement. The spec explicitly allows servers to describe *why* a request is forbidden in the response body if they choose to be transparent.
