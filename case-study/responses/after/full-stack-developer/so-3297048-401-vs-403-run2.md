## 401 vs 403: The Spec-Based Distinction

The core axis is **identity vs. permission**.

### 401 Unauthorized — "Prove who you are"

Per **RFC 9110 §15.5.2**, 401 means the request lacks valid *authentication credentials*. Despite its name, it really means **unauthenticated** (the name is a historical accident).

**Key requirement:** the server **MUST** include a `WWW-Authenticate` header telling the client *how* to authenticate (e.g., `WWW-Authenticate: Bearer realm="api"`). The implicit contract is: *"I don't know who you are — tell me, and maybe you'll get in."* The client is expected to retry with credentials.

### 403 Forbidden — "You're known, and still can't have it"

Per **RFC 9110 §15.5.4**, 403 means the server understood the request but *refuses to fulfill it* — and identity is already established. No `WWW-Authenticate` header is expected. The contract: *"I know who you are, and you still can't have this."* Retrying with different credentials won't help; you need different *permissions*.

---

### The Actor Model

| | 401 | 403 |
|---|---|---|
| Server's knowledge of caller | Nothing / invalid creds | Identity established |
| What failed | Authentication (proving identity) | Authorization (having permission) |
| Can retry fix it? | Yes, with valid credentials | No (unless permissions change) |
| Analogy | "Show me your ID" | "I see your ID — you're not on the list" |
| Required header | `WWW-Authenticate` (MUST) | None |

401 is always logically "earlier in the pipeline" — you can't authorize someone you haven't authenticated.

---

### What to serve in your concrete scenarios

| Scenario | Status | Rationale |
|---|---|---|
| Not logged in (no credentials) | **401** | Identity unconfirmed |
| Session expired / invalid token | **401** | Same — identity unconfirmed |
| Logged in, wrong group/role | **403** | Identity confirmed, permission denied |

---

### The 404 Escape Hatch (Information Leakage)

RFC 9110 §15.5.4 explicitly says: *"If the server does not wish to make that information available to the client, it can instead use 404 (Not Found)."*

A 403 confirms the resource **exists**. Whether that's acceptable depends on your threat model:

- **Resource existence is public knowledge** → 401/403 are fine
- **Existence itself is sensitive** (e.g., a secret admin path, another tenant's data) → return **404** to prevent enumeration

GitHub does this for private repos: unauthenticated requests get 404, not 403, so you can't confirm whether a private repo exists.

---

### Bottom Line

- Not logged in → **401** (+ `WWW-Authenticate`)
- Logged in, wrong group → **403**
- Resource existence must stay hidden → **404** (for both cases)

Authoritative source: **RFC 9110 §15.5.2** (401) and **§15.5.4** (403), which supersede the older RFC 7235 / RFC 7231.
