## 401 vs 403: The Actor Model

The names are a famous historical accident:
- **401 Unauthorized** semantically means **unauthenticated**
- **403 Forbidden** semantically means **unauthorized**

---

### RFC 9110 (the current spec, supersedes RFC 7235)

**401** — *"I don't know who you are. Identify yourself and try again."*
- Signals that valid authentication credentials are **missing or invalid**
- The server **MUST** send a `WWW-Authenticate` header telling the client *how* to authenticate
- The response is **retryable**: supplying correct credentials may change the outcome
- Without something meaningful to put in `WWW-Authenticate`, 401 is the wrong code

**403** — *"I know who you are (or it doesn't matter), and the answer is no."*
- The server understood the request fully and refuses it **regardless of credentials**
- RFC 9110 explicitly notes: if credentials were provided, the server considers them **insufficient** — the client `SHOULD NOT` repeat with the same credentials
- Re-authentication is not the remedy
- The resource's existence is implicitly acknowledged

---

### The Pivotal Question

> **Would supplying or correcting authentication credentials plausibly change the outcome?**

| Scenario | Code |
|---|---|
| No credentials, and credentials are required | **401** |
| Credentials present but invalid/expired (retryable) | **401** |
| Valid credentials, but wrong role/group | **403** |
| Resource is categorically off-limits regardless | **403** |
| You want to hide the resource's existence entirely | **404** *(RFC 9110 explicitly endorses this)* |

---

### Practical Recommendation

- **Unauthenticated user hits a protected page** → **401** (browser-facing apps often use a login redirect instead, which is pragmatic but not strictly RFC-compliant)
- **Authenticated user lacks the required role/group membership** → **403**, unambiguously
- **You want to hide that the resource exists** → **404**

The most common mistake is returning **403 to unauthenticated users** on the grounds that "they're definitely not allowed." This is only correct if you mean authentication is entirely irrelevant — for most apps it incorrectly forecloses the client's ability to know that logging in would help.

The `WWW-Authenticate` requirement on 401 is load-bearing: it is not a formality. A 401 without it is technically malformed per RFC 9110.
