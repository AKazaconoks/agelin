## 401 vs 403: The Spec-Based Distinction

### One-sentence actor model

**401** = "I don't know who you are — identify yourself and try again."
**403** = "I know who you are (or it doesn't matter) — you are not allowed, full stop."

---

### 401 Unauthorized (RFC 9110 §15.5.2)

Despite the name, 401 is about **authentication**, not authorization. The spec requires a `WWW-Authenticate` response header — its absence makes the response non-conformant.

> "The server generating a 401 response MUST send a WWW-Authenticate header field containing at least one challenge applicable to the target resource."

The implicit contract: **the server is willing to reconsider if credentials are supplied.** The door is open; the server is asking for a key.

---

### 403 Forbidden (RFC 9110 §15.5.4)

403 is about **authorization**. The server understood the request and has decided it will not fulfill it, and **authentication alone will not change the outcome**.

> "The 403 (Forbidden) status code indicates that the server understood the request but refuses to fulfill it."

The implicit contract: **the door is locked and your credential *category* will never open it.** Escalating to a different role *could* help; re-authenticating as the same identity will not.

---

### Your concrete scenario — two sub-cases

| Situation | Correct code |
|---|---|
| User is **not logged in** | **401** — server can't evaluate authorization yet; `WWW-Authenticate` tells the client how to authenticate |
| User is **logged in but wrong group/role** | **403** — identity is known, access is denied regardless of re-authentication |
| You want to reveal nothing about the resource's existence | **404** — RFC 9110 explicitly permits this |
| Credentials sent but expired/invalid | **401** — same as not logged in; re-auth may succeed |

---

### Common real-world deviation

Many frameworks collapse both unauthenticated and unauthorized into 403 to avoid leaking "this resource exists and requires auth." That is a deliberate security tradeoff against strict spec compliance — acceptable, but worth being aware of.
