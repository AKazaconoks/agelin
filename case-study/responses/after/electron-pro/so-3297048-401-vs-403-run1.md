## HTTP 401 vs 403: The Actor Model

The names are famously misleading. Here's the core distinction:

- **401** = *"I don't know who you are. Authenticate and try again."*
- **403** = *"I know who you are (or it doesn't matter). You're not getting in."*

"Unauthorized" in the HTTP sense actually means **unauthenticated**. "Forbidden" is what most people mean when they say "unauthorized."

---

### What RFC 9110 Says

**401 Unauthorized (§15.5.2):**
> "…the request has not been applied because it **lacks valid authentication credentials**… The server generating a 401 response **MUST send a `WWW-Authenticate` header** containing at least one challenge."

The `WWW-Authenticate` header requirement is load-bearing — 401 is a *negotiation prompt* telling the client *how* to authenticate.

**403 Forbidden (§15.5.4):**
> "…the server understood the request but refuses to fulfill it… **unlike 401, the server is indicating that authentication will not help**, and the request SHOULD NOT be repeated."

**"Authentication will not help"** is the RFC's own decision rule.

---

### The Two Cases

| Scenario | Code | Reason |
|---|---|---|
| User not logged in, hits protected page | **401** | Authentication *would* help — logging in might grant access |
| User logged in, wrong role/group | **403** | Auth cannot help — identity is known and refused |
| Resource existence must be hidden | **404** | Deliberate: GitHub does this for private repos to avoid leaking existence |

---

### Decision Rule

```
Missing valid credentials?
  YES → 401  (+ WWW-Authenticate header)
  NO  → Permitted?
          YES → 200
          NO  → 403
```

For browser apps with a login form rather than HTTP auth, some teams issue a **302 redirect** to `/login` instead of 401 — that's a UX convenience, not a semantically more correct choice.
