## 401 vs 403: The Actor Model

The names are confusing because **401 "Unauthorized" is actually about authentication**, not authorization. The RFC (9110 §15.5.2) is explicit: it means "I don't know who you are — identify yourself." It **must** be accompanied by a `WWW-Authenticate` header challenging the client to provide credentials.

**403 "Forbidden"** (RFC 9110 §15.5.4) means the server understood the request, knows (or doesn't care) who is asking, and has decided **no**. Critically, the spec says the client "SHOULD NOT automatically repeat the request with different credentials" — there is no challenge, no retry loop. The door is closed.

### The clean mental model

| Code | Server is saying | Client should |
|------|-----------------|---------------|
| **401** | "Prove who you are." | Re-issue with credentials (or show login UI) |
| **403** | "I know who you are. No." | Show an error; don't retry |

---

## Concrete Recommendations

**(a) User is not logged in (anonymous request)**
→ **Return 401** + `WWW-Authenticate` header. Your client-side code intercepts it and redirects to login. If the resource's *existence* is sensitive, return **404** instead — the spec explicitly permits this (§15.5.4) to prevent enumeration attacks.

**(b) User is logged in but lacks the required role/group**
→ **Return 403**. The identity is established; the authorization decision is negative and final. Returning 401 here would mislead the client into thinking re-authenticating will help, which it won't.

---

## When to use 404

A common secure pattern: return 404 to *unauthenticated* users (hides existence), but 403 to *authenticated* users who lack the role (they know the resource exists but are denied — useful UX, no enumeration risk since they're already identified).

---

## Decision Tree

```
Request arrives
│
├── Resource existence is sensitive?
│   └── YES + user is anonymous → 404
│
├── Valid credentials present?
│   ├── NO → 401 (+ WWW-Authenticate)
│   └── YES → identity has permission?
│           ├── YES → 200
│           └── NO  → 403
```

**Relevant specs:** RFC 9110 §§11, 15.5.2, 15.5.4, 15.5.5 (supersedes RFC 7235).
