## 401 vs 403: The Actor Model

### The Two-Phase Security Model

The distinction maps to two entirely different questions:

| Code | Question answered | Phase |
|------|-------------------|-------|
| **401** | *Who are you?* | **Authentication** |
| **403** | *Are you allowed?* | **Authorization** |

Despite its name, **401 "Unauthorized" actually means "unauthenticated"** — the naming is a historical mistake in the spec that has never been corrected.

---

### What RFC 9110 Says

**401 (§15.5.2):**
- The request lacks valid authentication credentials.
- The server **MUST** include a `WWW-Authenticate` header with at least one challenge (e.g., `WWW-Authenticate: Bearer realm="api"`).
- Semantics: "I don't know who you are — send credentials and I'll reconsider."

**403 (§15.5.4):**
- The server understood the request but refuses to fulfill it.
- **MUST NOT** include `WWW-Authenticate`.
- Semantics: "I know who you are, and the answer is still no."

---

### Applying This to Your Scenario

| Situation | Code | Reason |
|-----------|------|--------|
| User is not logged in | **401** | Identity unknown; credentials would change the outcome |
| User is logged in but lacks the required group | **403** | Identity known; higher privilege would be needed (re-authing won't help) |

---

### The 404 Masking Nuance

Both 401 and 403 implicitly confirm the resource exists. If the resource's existence is itself sensitive, returning **404** is a common and deliberate choice — you sacrifice helpfulness for non-disclosure. This is a security/UX trade-off, not a spec requirement.

---

### Bottom Line

> **401** = authentication problem (no identity established)  
> **403** = authorization problem (identity established, but insufficient)

The `WWW-Authenticate` header is the mechanical signal: 401 requires it (it tells the client *how* to authenticate); 403 omits it (authentication is irrelevant to the outcome).
