## The actor model

The key question is: **does the server know who the client is?**

| Code | Server's knowledge | What the server is saying |
|---|---|---|
| **401** | Identity unknown/unverified | "I don't know who you are — please authenticate" |
| **403** | Identity known, permissions lacking | "I know who you are, but you can't access this" |

---

## 401 Unauthorized

**RFC 9235 §3.1**: "The 401 status code indicates that the request has not been applied because it **lacks valid authentication credentials** for the target resource."

- **MUST** include a `WWW-Authenticate` header with at least one challenge (e.g. `WWW-Authenticate: Bearer realm="api"`)
- The implicit message to the client: "retry with valid credentials and you may get in"
- Use when: no credentials were provided, or the provided credentials are invalid/expired

## 403 Forbidden

**RFC 9110 §15.5.4**: "The server understood the request but **refuses to fulfill it**… re-authenticating will make no difference."

- Should **not** include `WWW-Authenticate`
- The implicit message: "even with valid credentials, you won't get access to *this* resource"
- Use when: the user is authenticated but belongs to the wrong group / lacks the required permission

---

## Applied to your scenario

| Situation | Correct response |
|---|---|
| User is **not logged in** | **401** + `WWW-Authenticate` header |
| User is logged in but **lacks the required privilege/group** | **403** |

---

## The common mistake

Many frameworks (and developers) reach for **403 for all access denials**, including unauthenticated requests. This is technically wrong: it tells the client that re-authenticating won't help — which is false when the user simply hasn't logged in yet.

The specs are unambiguous: **401 = authentication problem** (who are you?), **403 = authorization problem** (what can you do?).
